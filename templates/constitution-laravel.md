# Project Constitution

## Core Principles

### I. Laravel Best Practices

All code MUST follow Laravel framework conventions and best practices.
This includes but is not limited to:

- Use Artisan generators (`php artisan make:*`) to create files
- Follow Eloquent conventions for models, relationships, and query scoping
- Use Form Requests for validation, Policies for authorization
- Prefer named routes and the `route()` helper for URL generation
- Use Eloquent API Resources for API responses
- Run `vendor/bin/pint --dirty --format agent` after modifying PHP files
- The `/laravel-best-practices` skill MUST be activated for every PHP code change

### II. Feature Tests by Default

Every code change MUST include feature tests written with Pest.
Factories MUST exist for every Eloquent model in the application.

- Feature tests are the default test type; unit tests are used only when
  testing isolated logic with no framework dependencies
- Use `php artisan make:test --pest` to create tests
- Factories MUST use realistic fake data and define states for common scenarios
- Tests MUST run green before any code is considered complete
- Use `RefreshDatabase` trait in feature tests
- Red-Green-Refactor: write failing tests first, then implement

### III. Brain Architecture

All business logic MUST flow through the Brain pattern
(Processes, Tasks, and Queries).

- **Processes** orchestrate sequences of Tasks within a DB transaction
- **Tasks** are single units of work that mutate state; always return `$this`
- **Queries** are read-only operations; NEVER mutate state inside a Query
- Controllers MUST remain thin — delegate to Processes/Tasks/Queries
- Tasks are reusable across Processes; design them as independent units
- Use `@property-read` docblocks to document expected payload shape
- Side-effects (emails, API calls) belong in queueable Tasks
<!-- [/CONDITIONAL:brain] -->

### IV. Livewire-First UI with Flux

All dynamic UI MUST be built with Livewire and Flux UI components.
Always try Flux components first before building custom markup.

- Check available `<flux:*>` components before writing custom HTML
- Use Flux for buttons, inputs, modals, forms, tables, badges, tooltips,
  date-pickers, and all standard UI elements
- Keep state server-side in Livewire components
- Use Alpine.js only for lightweight client-side interactions that do not
  require server state
- No JavaScript frameworks (React, Vue, etc.) unless explicitly approved
- Validate and authorize in Livewire actions as you would in controllers
<!-- [/CONDITIONAL:flux] -->

### V. Simplicity

Favor the simplest solution that meets current requirements.
Do not build for hypothetical future needs.

- YAGNI: do not add features, abstractions, or configurability beyond
  what is explicitly required
- Three similar lines of code are better than a premature abstraction
- Do not create helpers, utilities, or wrappers for one-time operations
- Stick to existing directory structure; do not create new base folders
  without approval
- Do not change application dependencies without approval
- Prefer editing existing files over creating new ones

### VI. Observability

The application MUST provide meaningful visibility into its behavior
for debugging and monitoring.

- Use structured logging with appropriate log levels
- Meaningful error messages that identify the problem and context
<!-- [CONDITIONAL:brain] -->
- Brain process/task events MUST be logged when `brain.log` is enabled
<!-- [/CONDITIONAL:brain] -->
- Use Laravel's built-in exception handling; do not swallow exceptions
  silently
- Prefer actionable log entries over verbose noise
